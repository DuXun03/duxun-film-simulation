# FilmSim Plugin -- 代码质量审计报告

审计日期：2026-05-28
审计人：Senior Developer (高级开发工程师)
审计范围：Phase 1 全部 5 个 DCTL 着色器（777 行）

---

## 一、总体评价

**当前等级**: B+（有潜力的技术原型，离生产就绪还有距离）

**亮点：**
- 色彩科学基础扎实，Cineon、Beer-Lambert、减色模型理解到位
- 代码注释详尽，色彩科学背景说明为团队知识传承建立了良好基础
- 整合方案文档（integration-plan.md）开源选型专业，MIT协议安全意识强
- 5种胶片Stock矩阵参数调校有审美判断力

**主要短板：**
- **零 UI 参数暴露** -- 所有参数都是 `const`，用户无法在 Resolve 面板调节
- **严重代码重复** -- Core 与 4 个独立模块存在大量重复逻辑
- **模块间行为不一致** -- Core 的曲线/密度公式与独立模块不同
- **缺少工程基础设施** -- 无共享库、无测试、无版本管理

## 三、跨模块问题

### 架构层面

```
当前 (Phase 1):
┌───────────┐   ┌──────────────┐   ┌────────────┐   ┌───────────┐   ┌───────┐
│ Contrast  │   │   Matrix     │   │  Density   │   │    CMY    │   │ Core  │
│  (独立)   │   │   (独立)     │   │  (独立)    │   │  (独立)   │   │(重复) │
│ 114行     │   │  144行       │   │  136行     │   │  142行    │   │241行  │
└───────────┘   └──────────────┘   └────────────┘   └───────────┘   └───────┘
      ↑               ↑                ↑                ↑              │
      └───────────────┴────── 全部重复实现 ──────────────┴──────────────┘
```

**根本问题**: Core 是 4 个独立模块的复制粘贴拼接版，而非通过调用组合。任何修复需要改两个地方。

**目标架构**:
```
┌──────────────────────────────────────────────────┐
│              FilmSim-Common.h                     │
│  _film_s_curve()  _film_lum()  _safe_log2() ... │
└──────────────────────────────────────────────────┘
          ↑         ↑         ↑         ↑
    ┌─────┴──┐ ┌───┴───┐ ┌───┴────┐ ┌──┴───┐
    │Contrast│ │Matrix │ │Density │ │ CMY  │  ← 独立模块(带UI参数)
    └───┬────┘ └──┬────┘ └───┬────┘ └──┬───┘
        └─────────┴──────────┴─────────┘
                       │
                 ┌─────┴─────┐
                 │ FilmSim   │ ← Core 通过 include 调用组合
                 │  -Core    │
                 └───────────┘
```

### 其他跨模块问题

| # | 问题 | 严重度 |
|---|------|--------|
| 1 | 无共享数学库 -- 5个文件各自实现 log2f/powf 安全包装 | P1 |
| 2 | 无 `DEFINE_UI_PARAMS` 统一模板 | P0 |
| 3 | 参数命名不一致 -- 同名参数在不同文件取值范围不同 | P2 |
| 4 | 无测试素材或验证脚本 | P1 |
| 5 | 缺少 .gitignore（DCTL编译产物 .dctlcc 应忽略） | P2 |

## 五、Phase 1.5 就绪度评估

**就绪度**: 70%

在开始移植 mycolor GLSL 代码之前，至少需要完成 P0 项。原因：
- 新代码若无 UI 参数，同样不可用
- 公共数学库必须在新增模块前建立，否则重复问题会成倍恶化
- 当前 Core 的架构不支持干净地插入新管线阶段

**建议执行顺序**:
1. 先建 FilmSim-Common.h（一次）
2. 修复 P0 Bug 并添加 UI 参数（一次）
3. 重构 Core 为组合模式（一次）
4. 然后启动 Phase 1.5 移植（新代码基于干净架构）
